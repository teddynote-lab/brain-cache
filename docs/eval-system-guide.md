# LLM 평가 시스템 구축 가이드

LLM 기반 애플리케이션의 품질을 체계적으로 측정하고 개선하기 위한 평가(Eval) 시스템 구축 가이드입니다.

## 왜 평가 시스템이 필요한가

- 프롬프트 수정 후 **기존 기능이 깨지는 것**(regression)을 방지
- "느낌"이 아닌 **데이터 기반**으로 개선 여부 판단
- 고비용 모델 대신 저비용 모델로 교체 가능 여부를 **객관적으로 검증**
- 팀원 간 품질 기준 통일

## 평가 시스템 설계 3단계

### 1단계: 데이터셋 구축

수동으로 20~50개의 입출력 쌍을 만드는 것부터 시작합니다.

```yaml
# eval_dataset.yaml
- input: "RAG에서 리랭킹의 역할은?"
  expected: "검색 결과를 정밀도 높은 모델로 재정렬하여 상위 결과의 품질을 높이는 것"
  tags: [rag, retrieval]
  
- input: "LoRA와 QLoRA의 차이점은?"
  expected: "QLoRA는 LoRA에 4bit 양자화를 결합하여 메모리 사용량을 줄인 방법"
  tags: [fine-tuning, efficiency]
```

**팁:**
- 실제 사용자 질문에서 추출하는 것이 가장 좋음
- 쉬운 케이스와 어려운 케이스를 균형있게 구성
- 엣지 케이스(빈 입력, 관련 없는 질문 등) 포함

### 2단계: 채점기 구축

#### 코드 기반 채점기

결정적으로 판단 가능한 항목에 사용합니다.

```python
def check_format(output: str) -> bool:
    """JSON 형식으로 응답했는지 확인"""
    try:
        json.loads(output)
        return True
    except:
        return False

def check_contains_source(output: str) -> bool:
    """출처 인용이 포함되어 있는지 확인"""
    return bool(re.search(r'\[출처\]|\[Source\]|참고:', output))
```

#### LLM-as-Judge 채점기

뉘앙스가 필요한 평가에 사용합니다.

```
아래 AI 응답을 평가하세요.

## 질문
{question}

## 정답 (참고용)
{expected}

## AI 응답
{actual}

## 평가
이 응답이 정확하고 유용한가요?
- PASS: 핵심 정보가 정확하게 포함됨
- FAIL: 핵심 정보가 누락되거나 잘못됨

판단: PASS 또는 FAIL
근거: (한 줄)
```

**캘리브레이션**: 사람이 채점한 20개 이상의 예시와 LLM 채점 결과를 비교하여 일치율 확인

### 3단계: CI/CD 통합

```
코드/프롬프트 변경
  → 코드 기반 채점기 실행 (CI, 빠르고 저비용)
  → LLM 채점기 실행 (Staging, 느리지만 정밀)
  → 모든 게이트 통과 시 배포
```

## 평가 유형

| 유형 | 시점 | 목적 |
|------|------|------|
| **Offline** | 배포 전 | 변경사항 검증 |
| **Online** | 프로덕션 | 실시간 품질 모니터링 |
| **Ad-hoc** | 수시 | 특정 실패 패턴 분석 |

## 도구

| 도구 | 특징 |
|------|------|
| **LangSmith** | 트레이스 + 평가 + 데이터셋 관리 통합 |
| **RAGAS** | RAG 특화 오픈소스 평가 |
| **Promptfoo** | 프롬프트 A/B 테스트 |
| **자체 구축** | 완전한 커스터마이징, pytest 기반 |

## 흔한 실수

1. **eval 없이 프롬프트 수정**: "느낌상 좋아졌다"는 증거가 아님
2. **범용 메트릭만 사용**: "helpfulness 4.2점"보다 구체적 실패 유형 추적이 유용
3. **데이터셋 고정**: 프로덕션에서 새 실패 패턴이 나오면 데이터셋에 추가
4. **모든 것을 자동화**: 에러 분석의 60~80%는 수동으로 트레이스를 읽는 것
