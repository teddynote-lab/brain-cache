# 프롬프트 엔지니어링 가이드

LLM을 활용한 애플리케이션에서 안정적이고 일관된 결과를 얻기 위한 프롬프트 설계 원칙을 정리합니다.

## 기본 원칙

### 1. 명확한 역할과 목표 설정

```
# Bad
"이 텍스트를 요약해줘"

# Good
"당신은 기술 문서 편집자입니다.
아래 텍스트를 AI 엔지니어가 3분 내에 핵심을 파악할 수 있도록
3~5줄의 bullet point로 요약하세요."
```

### 2. 구조화된 출력 지정

출력 형식을 명시하면 파싱이 쉽고 일관성이 높아집니다.

```
다음 JSON 형식으로 응답하세요:
{
  "summary": "한 줄 요약",
  "key_points": ["포인트1", "포인트2"],
  "confidence": 0.0~1.0
}
```

### 3. Few-shot 예시 활용

추상적인 규칙보다 구체적인 입출력 예시가 효과적입니다.

```
## 예시

입력: "GPT-4는 2023년에 출시된 OpenAI의 모델이다"
출력: {"entity": "GPT-4", "org": "OpenAI", "year": 2023}

입력: "Claude 3.5 Sonnet은 Anthropic이 만들었다"
출력: {"entity": "Claude 3.5 Sonnet", "org": "Anthropic", "year": null}
```

### 4. Chain-of-Thought

복잡한 추론이 필요할 때 단계별로 생각하도록 유도합니다.

```
문제를 풀기 전에 먼저 접근 방법을 설명하고,
단계별로 풀이 과정을 보여준 후,
최종 답변을 제시하세요.
```

## 실무 패턴

### 분류 (Classification)

```
아래 텍스트를 다음 카테고리 중 하나로 분류하세요.
반드시 아래 카테고리 중 하나만 선택하세요:
- Bug Report
- Feature Request
- Question
- Other

분류할 수 없는 경우 "Other"를 선택하세요.
```

### 정보 추출 (Extraction)

```
아래 계약서에서 다음 정보를 추출하세요.
문서에 명시되지 않은 항목은 null로 표시하세요:
- 계약 당사자
- 계약 기간
- 계약 금액
- 해지 조건
```

### 평가 (LLM-as-Judge)

```
아래 AI 응답의 품질을 평가하세요.

## 평가 기준
- 정확성: 사실에 부합하는가 (1~5)
- 관련성: 질문에 대한 답변인가 (1~5)
- 완전성: 필요한 정보를 포함하는가 (1~5)

각 기준에 대해 점수와 근거를 제시하세요.
```

## 안티패턴

| 안티패턴 | 문제점 | 개선 방법 |
|---------|--------|----------|
| 모호한 지시 | 결과가 매번 다름 | 구체적 기준과 형식 명시 |
| 과도한 제약 | 유용한 답변 생성 어려움 | 핵심 제약만 유지 |
| 컨텍스트 과잉 | 중요 정보가 묻힘 | 관련 정보만 선별 |
| 네거티브 지시만 | 무엇을 해야 하는지 불명확 | "~하지 마" 대신 "~해라" |

## 프롬프트 버전 관리

프롬프트도 코드처럼 버전 관리가 필요합니다.

- 프롬프트 변경 시 **eval 세트로 regression 체크**
- 변경 이유와 결과를 기록
- A/B 테스트로 개선 여부를 정량적으로 확인
