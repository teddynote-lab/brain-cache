# RAG 파이프라인 구축 가이드

Retrieval-Augmented Generation(RAG) 시스템을 설계하고 구축할 때 고려해야 할 핵심 요소들을 정리합니다.

## 전체 아키텍처

```
문서 수집 → 청킹 → 임베딩 → 벡터 DB → 검색 → 리랭킹 → LLM 생성 → 응답
```

## 1. 문서 처리 (Ingestion)

### 청킹 전략

| 전략 | 적합한 경우 | 주의점 |
|------|-----------|--------|
| 고정 크기 | 구조가 없는 일반 텍스트 | 문맥이 잘릴 수 있음 |
| 시맨틱 청킹 | 주제가 자주 전환되는 문서 | 임베딩 모델 추가 비용 |
| 구조 기반 | 마크다운, HTML 등 구조화 문서 | 헤딩 기준으로 자연스러운 분할 |
| 재귀적 분할 | 범용적 활용 | 오버랩 크기 튜닝 필요 |

### 실무 팁

- 청크 크기는 **512~1024 토큰**이 일반적인 시작점
- 오버랩은 청크 크기의 **10~20%**
- 테이블, 이미지 캡션은 별도 처리 필요
- 메타데이터(출처, 날짜, 섹션)를 청크에 포함시키면 필터링에 유리

## 2. 임베딩 & 벡터 DB

### 임베딩 모델 선택 기준

- **다국어 지원**: 한국어 포함 시 multilingual 모델 필수
- **차원 수**: 높을수록 정밀하지만 저장/검색 비용 증가
- **MTEB 벤치마크**: retrieval task 성능 확인

### 벡터 DB 선택

| DB | 특징 | 적합한 규모 |
|----|------|-----------|
| Chroma | 경량, 로컬 개발용 | 소규모 PoC |
| Pinecone | 완전 관리형, 빠른 시작 | 중소규모 프로덕션 |
| Weaviate | 하이브리드 검색, 필터링 강점 | 중대규모 |
| Milvus | 대규모 최적화, 자체 호스팅 | 대규모 프로덕션 |

## 3. 검색 & 리랭킹

### 하이브리드 검색

키워드 검색(BM25)과 시맨틱 검색(벡터)을 결합하면 단독 사용 대비 높은 recall을 얻을 수 있습니다.

```
최종 스코어 = α × 시맨틱 스코어 + (1-α) × BM25 스코어
```

- α는 보통 **0.5~0.7**에서 시작하여 데이터에 맞게 조정

### 리랭킹

Top-K 결과를 Cross-Encoder로 재정렬하면 precision이 크게 향상됩니다.

- 검색: 빠르고 넓게 (Top 20~50)
- 리랭킹: 정밀하게 (Top 3~5로 축소)

## 4. 프롬프트 & 생성

### 프롬프트 구조

```
[시스템 프롬프트: 역할, 규칙]
[검색된 컨텍스트: 문서 청크들]
[사용자 질문]
```

### 핵심 원칙

- 검색 결과에 **없는 내용은 생성하지 않도록** 지시
- 출처 인용을 요구하면 hallucination 감소
- 컨텍스트가 부족하면 **"모르겠다"고 답하도록** 유도

## 5. 평가

RAG 시스템 평가는 검색과 생성을 분리해서 측정합니다.

### 검색 평가

| 지표 | 의미 |
|------|------|
| Recall@K | 정답 문서가 Top-K에 포함되는 비율 |
| MRR | 정답 문서의 평균 역순위 |
| NDCG | 순위 품질 종합 점수 |

### 생성 평가

| 지표 | 의미 |
|------|------|
| Faithfulness | 검색 결과에 근거한 답변인가 |
| Relevance | 질문에 대한 답변이 적절한가 |
| Completeness | 필요한 정보를 빠뜨리지 않았는가 |

### 도구

- **RAGAS**: 오픈소스 RAG 평가 프레임워크
- **LangSmith**: LangChain 생태계 평가/모니터링
- **직접 구축**: LLM-as-judge로 커스텀 평가기 구성
